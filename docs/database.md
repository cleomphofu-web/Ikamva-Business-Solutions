## Tables

### tenants

Stores account boundaries for the operating system.

Important fields:

- `id`
- `name`
- `slug`
- `status`
- `created_at`
- `updated_at`

### tenant_users

Maps auth-provider users into tenants.

Important fields:

- `tenant_id`
- `user_id`
- `email`
- `full_name`
- `role`
- `status`

### client_profiles

Stores client-level operating and quota state.

Important fields:

- `monthly_task_limit`
- `tasks_used_this_month`
- `billing_cycle_start`
- `billing_cycle_end`

### client_sops

Stores versioned task instructions and model settings.

Important fields:

- `version`
- `active`
- `system_prompt`
- `validation_schema`
- `output_schema`
- `model_provider`
- `model_name`
- `model_settings`

Only one active SOP is allowed per tenant and task type.

### task_queue

Durable task state machine.

Status values:

- `pending`
- `validating`
- `waiting_quota`
- `processing`
- `awaiting_human`
- `completed`
- `failed`
- `cancelled`
- `quota_exceeded`

Important fields:

- `tenant_id`
- `sop_id`
- `task_type`
- `payload`
- `normalized_payload`
- `status`
- `priority`
- `idempotency_key`
- `retry_count`
- `locked_at`
- `locked_by`
- `scheduled_for`

### task_logs

Immutable task history. Inserts are generated by the task queue status trigger. Updates and deletes are blocked by trigger.

Every task creation and status transition produces a log entry.

## Provider Independence

The schema is Supabase-compatible Postgres, but application code must reach it through repositories and database service adapters.
